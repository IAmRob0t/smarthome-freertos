# 基于 FreeRTOS 的室内环境远程监测与控制系统

# Indoor Environmental Remote Monitoring and Control System Based on FreeRTOS

---

## 摘要

面向室内环境监测与远程设备控制场景，本文设计并实现了一套基于 FreeRTOS 的嵌入式系统。系统以 STM32F103ZET6 为主控，集成 DHT11 温湿度采集、LM393 光照状态检测、MQ-135 空气质量采集、ESP8266 无线通信及 OLED 信息显示模块，并采用 PC 图形界面工具完成数据监控与远程控制闭环验证。与传统“单循环+阻塞式”实现方式相比，本系统重点采用“任务化、分层化、事件化”的软件工程方案：以 FreeRTOS 为调度核心，以输入事件队列为信息中轴，以 JSON 命令作为统一控制参数，实现采集、通信、控制、显示等功能模块解耦。

本文重点围绕软件开发流程与框架设计展开：首先完成需求拆解与子系统抽象，其次建立驱动层到业务层的分层结构，再进行任务优先级规划与通信协议实现，最后通过联调测试验证系统可用性。实验结果表明：系统可稳定完成环境数据周期上传，能够实时接收并执行远程控制命令，具备良好的可维护性与可扩展性。本文所采用的软件架构适用于课程设计、毕业设计及中小型物联网控制系统的快速落地。

**关键词**：FreeRTOS；STM32F103ZET6；软件架构；任务调度；UDP 通信；智能家居

## Abstract

This paper presents the design and implementation of an indoor environmental remote monitoring and control system based on FreeRTOS. The system is built on an STM32F103ZET6 microcontroller and integrates DHT11 temperature-humidity sensing, LM393 light-state detection, MQ-135 air-quality acquisition, ESP8266 wireless communication, and OLED local display. A PC-based graphical UDP tool is used to verify the complete closed loop of real-time monitoring and remote control.

The core contribution of this work is the software-oriented engineering method rather than hardware complexity. A task-based, layered, and event-driven framework is adopted. FreeRTOS serves as the scheduling kernel; an input-event queue is designed as the central information bus; and JSON is used as a unified command format for control abstraction. This architecture decouples sensing, communication, control, and display modules, significantly improving maintainability and scalability.

The development process includes requirement decomposition, subsystem abstraction, layered implementation, task-priority planning, protocol integration, and end-to-end debugging. Experimental results show that the system can stably upload sensor data periodically and execute remote control commands in real time. The proposed framework is suitable for graduation projects and small-to-medium embedded IoT systems.

**Keywords**: FreeRTOS; STM32F103ZET6; Software Architecture; Task Scheduling; UDP Communication; Smart Home

# 第 1 章 绪论

## 1.1 研究背景与意义

随着物联网应用向家庭场景持续渗透，用户对环境可视化、设备远程控制、系统实时响应等能力提出了更高要求。传统裸机程序在功能较少时实现简单，但当系统同时承担传感采集、网络通信、界面显示与执行控制等任务时，常出现如下问题：

1. 程序耦合严重，功能扩展成本高；
2. 轮询结构阻塞明显，实时性不可控；
3. 模块边界不清晰，联调与维护困难；
4. 控制协议和数据格式分散，接口复用能力弱。

因此，在毕业设计中引入 FreeRTOS 与分层架构思想，采用工程化的软件方法构建完整系统，具有显著的教学与实践价值：不仅能实现“功能可用”，更能实现“结构清晰、可维护、可扩展”的目标。

## 1.2 课题目标

本课题目标并非追求复杂硬件堆叠，而是构建一套可落地的、软件结构合理的室内环境监测与控制系统。具体目标如下：

1. 实现温湿度、光照状态、空气质量等环境参数采集；
2. 实现通过 Wi-Fi 的数据上报与远程指令下发；
3. 实现本地执行器（灯、风扇）控制与状态联动；
4. 构建基于 FreeRTOS 的多任务框架，保证系统实时性；
5. 形成可复用的软件架构与调试流程文档。

## 1.3 本文主要工作

围绕上述目标，本文主要完成了以下工作：

1. 基于课程资料（见 `resource/note`）完成系统分层与任务框架设计；
2. 结合现有开发板与扩展板，完成传感器与网络模块接入；
3. 基于事件队列与 JSON 机制实现统一输入控制路径；
4. 完成 ESP8266 AT 指令链路、UDP 双向通信与 GUI 联调；
5. 完成系统功能验证，并沉淀问题定位与修复过程。

# 第 2 章 系统需求分析与总体方案

## 2.1 功能需求分析

根据项目任务清单与演示场景，系统需满足以下核心需求：

1. **环境监测需求**：周期输出温度（T）、湿度（H）、光照状态（L）与空气质量原始值（AQ）；
2. **远程控制需求**：接收 JSON 指令并控制灯、风扇及“回家/离家”联动；
3. **实时显示需求**：支持串口实时输出与 PC 图形界面监控；
4. **通信可靠需求**：支持 Wi-Fi 自动连接、UDP 通道建立、发送接收可验证；
5. **工程可维护需求**：软件具备模块边界，便于后续增加新传感器或新控制对象。

## 2.2 非功能需求

1. **实时性**：关键采集任务周期稳定；
2. **可扩展性**：新增输入设备或控制对象时尽量不改动核心业务逻辑；
3. **可测试性**：支持分层调试（驱动测试、网络测试、业务联调）；
4. **可读性**：模块命名与接口语义明确，便于毕业设计答辩说明。

## 2.3 总体技术方案

系统采用“感知层—控制层—网络层—应用层”四层架构：

1. **感知层**：DHT11、LM393、MQ-135；
2. **控制层**：STM32F103ZET6 + FreeRTOS 任务调度；
3. **网络层**：ESP8266（AT 指令 + UDP）；
4. **应用层**：PC 端 `udp_gui.py` 实时监控与远程控制。

该方案避免了复杂云平台依赖，适合答辩现场稳定演示，也符合“先打通本地实时链路，再逐步扩展”的工程实践路径。

## 3.1 主控与外围组成

系统主控芯片为 **STM32F103ZET6**（已修正型号），核心频率 72 MHz，具备丰富 GPIO、USART、定时器等资源，能够满足本课题多任务调度与多外设并发访问需求。

外围模块包括：

1. DHT11 温湿度传感器；
2. LM393 光照模块（DO 状态输入 + AO 采样通道）；
3. MQ-135 空气质量传感器（AO 采样）；
4. ESP8266 Wi-Fi 模块；
5. OLED 显示模块；
6. LED、风扇等执行器。

# 第 4 章 软件设计（重点）

## 4.1 软件开发流程

本课题采用分阶段迭代的软件开发流程：

1. **需求拆解阶段**：将“监测+上报+控制”拆解为可验证功能点；
2. **框架设计阶段**：按课程方法将系统拆分为输入、显示、网络、设备、字体、业务六个子系统；
3. **驱动接入阶段**：完成 DHT11、LM393/MQ-135、ESP8266、OLED 的基础驱动与初始化；
4. **任务化实现阶段**：创建 SmartHomeTask 与采集任务，建立周期采集和事件处理链路；
5. **协议联调阶段**：统一 JSON 控制协议，完成 UDP 双向通信调试；
6. **系统验证阶段**：按“串口输出—GUI 接收—按钮控制—闭环确认”完成验收。

该流程核心在于：每一阶段都有可观测输出，避免“到最后一次性联调失败”的风险。

## 4.2 软件架构分层设计

参考 `resource/note/05_项目1_基于HAL库的智能家居/01_程序功能及框架设计.md`，系统采用“子系统分层 + 结构体抽象”思想。当前工程目录体现了清晰层次：

1. `driver` / `cal` / `kal`：驱动与抽象适配层；
2. `device`：设备对象抽象层；
3. `input`：输入系统（按键/网络）统一入口；
4. `net`：网络设备与 AT 指令实现；
5. `smarthome`：业务逻辑层（任务、策略、控制）；
6. `unittest`：模块级测试支持。

这种设计使“硬件变化”与“业务逻辑”相对隔离。例如更换网络模块时，优先改动 `net` 子系统而不直接冲击业务层。

## 4.3 FreeRTOS 任务模型与调度策略

系统在 `main.c` 中创建核心任务：

1. `SmartHomeTask`：系统初始化、Wi-Fi 连接、网络输入事件处理、设备控制；
2. `DHT11_UpdateValue`：周期采集环境数据并上报。

调度策略设计要点：

1. 业务任务与采集任务并行，避免单任务阻塞；
2. 采集任务固定周期（当前 5 s）运行，保证数据更新节奏；
3. 网络输入由中断/回调驱动进入事件队列，提高控制响应性。

## 4.4 输入事件队列与 JSON 参数统一

参考 `resource/note/03_基于FreeRTOS实现智能家居/16_智能家居项目增加功能_框架设计.md` 与业务系统笔记，系统将“输入事件队列”作为信息中轴：

1. 按键输入、网络输入统一封装为 `InputEvent`；
2. 中间层 `ConvertInputEventToJson()` 将事件转换为统一 JSON 参数；
3. 业务层只关心 `{"dev":"xxx","status":"y"}`，不耦合输入来源。

该方法将“控制语义”从“输入来源”中剥离，提升了系统可扩展性：后续增加蓝牙、串口命令或新上位机时，不需要重写核心控制逻辑。

## 4.5 传感采集与数据上报实现

当前系统周期采集并输出：

1. `T`：温度；
2. `H`：湿度；
3. `L`：光照阈值状态（DO）；
4. `AQ`：空气质量原始值（ADC 采样）。

串口侧输出形如：

```text
T=23,H=56,L=0,A=1443
```

网络上报侧输出形如：

```json
{"T":23,"H":56,"L":0,"AQ":1443}
```

其中，`AQ` 字段用于在上位机界面区分“光照状态 L”与“空气质量模拟值”。

## 4.6 网络通信链路设计（ESP8266 + UDP）

网络模块采用 ESP8266 AT 指令实现，通信流程如下：

1. 设置工作模式：`AT+CWMODE=3`；
2. 连接热点：`AT+CWJAP="SSID","PASSWORD"`；
3. 建立 UDP 传输：`AT+CIPSTART="UDP",...`；
4. 发送数据：`AT+CIPSEND` + JSON 载荷；
5. 接收数据：解析 `+IPD` 帧并转为输入事件。

考虑实际网络抖动，本课题在实现中对控制类 AT 命令超时进行了工程化调整，降低了“通道建立失败（Create Transfer err）”概率，提高了联调稳定性。

## 4.7 上位机联调与交付工具设计

为替代不可用小程序端，项目采用 `resource/udp_gui.py` 作为交付工具，功能包括：

1. 监听 UDP 数据并显示实时日志（RX）；
2. 提供灯、风扇、回家/离家按钮（TX）；
3. 支持自定义 JSON 指令发送；
4. 支持时间戳日志，便于演示与问题定位。

经过联调，系统已形成可复现的双向链路：

- 上报链路：板端 -> GUI（RX）；
- 控制链路：GUI -> 板端（TX）。

# 第 5 章 系统实现与测试

## 5.1 实现结果

系统实现完成后，能够稳定达到以下状态：

1. 串口持续输出环境参数；
2. GUI 持续接收上报 JSON（`T/H/L/AQ`）；
3. GUI 按钮可远程控制灯、风扇及联动模式；
4. 网络重连后系统可恢复通信。

## 5.2 测试方案设计

测试采用“单元验证 + 闭环验证”两层结构。

### 5.2.1 单元验证

1. **采集单元**：验证 DHT11 与 ADC 采样连续输出；
2. **网络单元**：验证 ESP8266 入网、UDP 建链、收发命令；
3. **控制单元**：验证 JSON 指令解析后执行器状态变化。

### 5.2.2 闭环验证

1. 板端周期上报环境参数；
2. GUI 显示 RX 实时数据；
3. GUI 下发 TX 控制指令；
4. 板端执行控制并继续上报。

## 5.3 关键测试记录（示例）

| 测试项 | 观察结果 | 结论 |
|---|---|---|
| Wi-Fi 连接 | 串口出现 `Connect WIFI ok` | 通过 |
| UDP 建链 | 串口出现 `Create Transfer ok` | 通过 |
| 数据上报 | GUI 周期出现 `RX {"T":...,"H":...,"L":...,"AQ":...}` | 通过 |
| 远程控制 | 点击按钮后执行器状态变化，日志出现 TX | 通过 |
| 闭环运行 | 监测与控制可同时工作 | 通过 |

## 5.4 工程问题与修复总结

在联调阶段，系统曾出现若干典型问题，最终均完成修复：

1. **宏命名冲突问题**：`SSID` 宏与函数参数同名导致编译错误，已通过参数重命名解决；
2. **AT 命令超时问题**：`CIPSTART` 初始超时时间过短，导致建链失败，已增加控制命令超时；
3. **任务时序问题**：采集任务在网络设备注册前启动，导致上报空指针，已通过循环内延迟获取设备句柄修复。

上述问题与修复过程表明：软件框架清晰后，问题定位更快、修改影响范围更可控。

## 6.1 工作总结

本文完成了一套基于 FreeRTOS 的室内环境远程监测与控制系统设计与实现。与仅关注硬件拼接的实现方式不同，本课题将重点放在软件工程方法：通过分层架构、任务化调度、事件队列中轴与 JSON 参数统一机制，实现了系统的模块解耦与高可维护性。项目最终达成了“可采集、可上传、可远控、可演示”的完整闭环目标，并形成了较规范的开发与调试流程。
